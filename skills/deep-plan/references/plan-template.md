# Plan Document Template

Copy this structure into the plan output path (e.g. `docs/PLAN.md`). Every
section is required, in this order. Replace the guidance text; keep the
headings. Tag claims `VERIFIED` / `UNVERIFIED` / `CORRECTED` as defined in
SKILL.md.

---

```markdown
# Plan: <task title>

> Generated by deep-plan on <date>. Ground-truth source: <vendor path or docs URL + version>.

## 1. Context

Why this work exists, in at most 5 lines: the system, the intent, the
definition of done, and the explicit non-goals.

## 2. Findings: current state

Evidence-backed inventory from Phase 1. For each relevant component:

| Component | Status | Evidence |
|-----------|--------|----------|
| `src/Checkout/CartCalculator.php` | works | wired in `config/services.xml:41` |
| `admin settings "discount cap"` | half-wired | collected in `ui/Settings.vue:88`, never read |
| `src/Legacy/PriceHelper.php` | dead | no references outside itself |

## 3. Findings: framework ground truth

Vendor facts from Phase 2 that the design relies on. One row per mechanism:

| Mechanism | Fact | Source |
|-----------|------|--------|
| Rule engine | `RuleInterface::match()` receives `RuleScope` | `vendor/…/Rule/RuleInterface.php:18` |
| Event | `checkout.cart.processed` fires after collectors | docs v6.7 <URL> |

Include mechanisms that turned out NOT to exist — those are findings too.

## 4. Audit findings

Defects from Phase 3, worst first. Each entry:

- **[SEVERITY] Short title** — what is wrong, why it matters, `file:line`.
  Severity order: security > data loss > correctness > reliability > maintainability.

## 5. Corrections

Everything previously believed (by the user, prior reports, subagents, or
yourself) that turned out to be wrong, with the evidence that corrected it.
This section is mandatory. If it is empty, state explicitly why the research
still qualifies as adversarial.

## 6. Gap analysis

Numbered list from Phase 4. Each gap: classification (BROKEN / MISSING /
FIGHTS-THE-FRAMEWORK), one-line description, evidence links to sections 2–4.

## 7. Design decisions

One subsection per gap:

### Decision <N>: <title>
- **Options considered:** A (one line), B (one line)
- **Decision:** the ONE committed choice
- **Rationale:** complexity / architecture fit / framework blessing, citing section 3 evidence
- **Rejected because:** one line per rejected option

## 8. Phased roadmap

Dependency-ordered. Phase 0 is always shippable security/correctness fixes.
For EACH phase:

### Phase <N>: <name>  (effort: S/M/L)
- **Objective:** one line
- **Depends on:** Phase <M> / nothing
- **Entry criteria:** what must be true to start
- **Tasks:**
  - [ ] Concrete action naming the exact files/classes/migrations it touches
  - [ ] …
- **Exit criteria:** verifiable statements (a command passes, a behavior is
  observable) — never "works well"

## 9. Risk register

| Risk | Likelihood | Impact | Mitigation / rollback |
|------|-----------|--------|----------------------|
| Migration locks table on prod-size data | medium | high | run on replica first; batched migration |

## 10. Verification

**Automated** — exact copy-paste-runnable commands and what output means pass:

    bin/phpunit --filter CartCalculatorTest
    npm run typecheck

**Manual E2E** — numbered steps a human can follow, each with the expected
observable result.
```

---

## Quality bar for this document

- An implementer opens it and never needs to ask which file, what change, or
  how to verify.
- A reviewer can spot-check any claim by following its citation.
- A stakeholder reads sections 1, 8, and 9 and understands scope, order, and risk.
